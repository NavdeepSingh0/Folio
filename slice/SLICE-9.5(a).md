# Folio Architecture Notes (Slice 9.5)

## Purpose

This document explains the architectural reasoning behind Folio.

It is **not** an implementation guide.

It explains why the system is designed this way so future implementation decisions remain consistent.

---

# Design Philosophy

The original version of Folio transformed

Document

↓

Markdown

This was sufficient for generating notes, but every new AI feature (flashcards, MCQs, revision, semantic search, explain simpler, mind maps) required re-reading or re-generating the same document.

Instead of treating Markdown as the primary artifact, Folio now treats structured learning data as the source of truth.

Markdown becomes only one possible representation.

---

# Layer 1 — Documents

A Document is exactly what the user uploads.

Examples

* PDF
* PPTX
* Markdown
* DOCX (future)

Documents are immutable source material.

They are never edited directly.

Their responsibility is only to preserve the original information.

---

# Layer 2 — Learning Objects

Learning Objects are generated from documents.

A Learning Object represents one coherent educational concept inside one document.

Examples

* Definition
* Process
* Algorithm
* Formula
* Comparison
* Example
* Code Explanation

Learning Objects are:

* editable
* versioned
* cacheable
* searchable
* renderable

Every downstream feature operates on Learning Objects instead of raw documents.

Examples

Flashcards

↓

Learning Objects

MCQs

↓

Learning Objects

Revision

↓

Learning Objects

Mindmaps

↓

Learning Objects

Explain Simpler

↓

Learning Objects

Knowledge Search

↓

Learning Objects

This separation prevents repeated document processing.

---

# Layer 3 — Canonical Concepts (Future)

Canonical Concepts are NOT implemented in this slice.

The schema reserves support for them.

Purpose:

Multiple Learning Objects from different documents may represent the same underlying idea.

Example

Lecture 31

↓

Paging

Lecture 35

↓

Virtual Memory Paging

Research Paper

↓

Demand Paging

These may eventually resolve into one Canonical Concept.

Canonical Concepts accumulate:

* aliases
* relationships
* mastery
* study priority

The schema must allow this later without requiring database redesign.

---

# Why Markdown is NOT the storage format

Markdown is presentation.

It is not knowledge.

Knowledge is stored as Learning Objects.

Markdown is generated by a renderer.

Future renderers include:

* Flashcards
* MCQs
* Revision Sheets
* Mind Maps
* HTML
* PDF
* Mobile Views

This prevents every feature from depending on Markdown parsing.

---

# Renderer Layer

Renderer responsibilities:

Input:

Learning Objects

Output:

Presentation format

Example:

MarkdownRenderer

↓

Markdown

Future renderers:

FlashcardRenderer

↓

Flashcards

MCQRenderer

↓

Questions

RevisionRenderer

↓

Revision Sheets

Renderer code never calls the LLM.

LLM code never formats Markdown.

The responsibilities remain completely separated.

---

# Two-Pass Generation

Single-pass generation asks the model to:

Understand

Organize

Explain

Format

Write Markdown

in one call.

This produces inconsistent quality.

Instead:

Pass 1

Input:

Entire topic

Output:

Structured outline

* concepts
* explanation order
* exam focus

Pass 2

Input:

Outline + topic

Output:

Learning Objects

The model now writes with awareness of the entire topic before producing explanations.

This more closely matches how humans write educational material.

---

# Ownership of Data

LLM owns:

* explanations
* examples
* exam tips
* titles

Code owns:

* IDs
* hashes
* keywords
* source references
* caching
* embeddings
* versions

Analytics owns:

* mastery score
* study priority

No field should have ambiguous ownership.

---

# Versioning

Learning Objects are versioned.

Regeneration creates a new version.

Stable identifiers remain unchanged.

This preserves annotations and future learning analytics.

---

# Caching

Every Learning Object has a content hash.

If content does not change:

Do not regenerate.

Reuse previous output.

This minimizes unnecessary inference.

---

# Future Expansion

The following features are intentionally deferred:

* Canonical Concept merging
* Confidence-based concept resolution
* Relationship graph
* Behaviour-driven mastery analytics
* Cross-document knowledge accumulation

The schema supports these features.

The implementation does not build them yet.

This keeps Slice 10 focused on improving note quality instead of building infrastructure prematurely.

---

# Guiding Principle

The LLM should only perform work that requires reasoning.

Everything deterministic should be implemented in code.

Reasoning:

* explanations
* educational rewriting

Code:

* formatting
* rendering
* caching
* indexing
* IDs
* storage
* analytics
* presentation
