# Slice 9.5 — Architecture Freeze & Generation Engine

Goal:
Freeze Folio's internal architecture before implementing the Output Engine.
This slice establishes the permanent data model, renderer abstraction, and
two-pass generation pipeline while intentionally deferring advanced knowledge
graph features until sufficient user data exists.

Estimated Time:
5–7 hrs

Risk:
MEDIUM

Screens involved:
- None (architecture slice)
- Existing generation pipeline only

---

## Why this slice exists

The project has evolved beyond a simple document → markdown converter.

Future features such as:

- Flashcards
- MCQs
- Revision Sheets
- Mind Maps
- Explain Simpler
- Knowledge Search
- Exam Workflow

must not depend directly on Markdown.

Markdown becomes only one presentation format.

This slice freezes the architecture so future slices build on a stable
foundation instead of changing the data model repeatedly.

---

## Build

### 1. Freeze the Core Data Model

Replace the current note representation with a Learning Object model.

Implement Pydantic models only.

Implement:

- Document
- LearningObject

Reserve (do not implement logic):

- CanonicalConcept
- RelationshipGraph

LearningObject becomes the primary storage model.

Markdown is no longer the source of truth.

---

### 2. Learning Object Schema

Every Learning Object must contain:

Required:

- id
- stable_id
- document_id
- topic_label
- version
- content_hash

LLM Generated:

- title
- definition
- explanation
- example
- exam_tip

Code Generated:

- keywords
- source_slides
- created_at
- updated_at

Reserved (nullable):

- concept_id
- related_ids
- embedding
- study_priority
- mastery_score

Every field must have one owner.

No field may have ambiguous ownership.

---

### 3. Stable Identifier System

Implement:

UUID → internal immutable ID

Slug → stable human-readable identifier

Example:

Internal ID

550e8400-e29b...

Stable ID

virtual-memory-paging

Titles may change.

Stable IDs must never change.

This prepares the project for future concept resolution.

---

### 4. Renderer Layer

Create a renderer abstraction.

Implement interface:

Renderer

Implement:

MarkdownRenderer

Create empty placeholders for:

- FlashcardRenderer
- MCQRenderer
- RevisionRenderer
- MindmapRenderer

Only MarkdownRenderer is functional.

Renderers must never call the LLM.

Renderers only transform Learning Objects into output formats.

---

### 5. Two-Pass Generation Pipeline

Replace:

Topic

↓

Single Generation

↓

Markdown

with:

Topic

↓

Planning Pass

↓

Structured Outline (JSON)

↓

Generation Pass

↓

Learning Objects

↓

Markdown Renderer

Planning Pass responsibilities:

- detect concepts
- determine explanation order
- identify exam focus

Generation Pass responsibilities:

- generate explanations
- examples
- exam tips

Do not generate Markdown directly.

Markdown must always be produced by MarkdownRenderer.

---

### 6. Structured Output

Generation Pass returns structured JSON.

Do NOT ask the model to format Markdown.

Each Learning Object should include:

{
    definition,
    explanation,
    example,
    exam_tip
}

MarkdownRenderer converts this into Markdown.

---

### 7. Token Budget

Apply explicit token limits.

Approximate targets:

Definition:
30–50 tokens

Explanation:
120–180 tokens

Example:
60–100 tokens

Exam Tip:
30–60 tokens

Prevent unnecessary verbosity.

---

### 8. Caching

Every Learning Object stores:

content_hash

Before regeneration:

If content_hash unchanged

↓

Reuse existing object

↓

Skip generation

Only regenerate modified Learning Objects.

---

### 9. Deferred Features

Do NOT implement:

- Canonical concept resolution
- Concept merging
- Confidence scoring
- Relationship graph
- Behaviour analytics
- Mastery engine
- Study priority engine

The schema must support these.

The implementation must NOT build them.

---

Definition of Done

✓ LearningObject becomes primary storage model

✓ MarkdownRenderer is the only renderer

✓ Two-pass generation is operational

✓ Planning pass outputs JSON outline

✓ Generation pass outputs Learning Objects

✓ Markdown is generated by renderer

✓ Token budgets implemented

✓ Stable IDs implemented

✓ Future architecture reserved but not implemented

---

Fallback

If two-pass generation exceeds the time budget:

Implement the renderer layer and Learning Object schema first.

Keep the existing generation call temporarily.

Two-pass generation may be completed in the following slice without requiring
schema changes.
