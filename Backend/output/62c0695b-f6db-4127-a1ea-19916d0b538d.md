## Counting Sort

**Definition:** A non-comparison-based sorting algorithm that sorts integers in linear time by counting occurrences of each value.

Counting sort operates by first counting the occurrences of each integer in the input array. These counts are then used to determine the positions of each element in the sorted output. The algorithm is particularly efficient when the range of input values (k) is not significantly larger than the number of elements (n). It is stable and works best with small integer ranges, such as sorting exam scores or ages. The process involves creating a count array, computing cumulative counts, and placing elements into their correct positions based on these counts. This method avoids direct comparisons between elements, achieving linear time complexity.

**Algorithm Steps:**
1. Create a count array with size equal to the range of input values.
2. Initialize all counts to zero.
3. Iterate through the input array and increment the count for each value.
4. Compute cumulative counts to determine the correct positions of each element.
5. Build the output array by placing elements in their correct positions based on cumulative counts.

```python
def counting_sort(arr):
    max_val = max(arr)
    count = [0] * (max_val + 1)
    for num in arr:
        count[num] += 1
    output = []
    for i in range(len(count)):
        output.extend([i] * count[i])
    return output
```

**Comparison:**
| Algorithm | Time Complexity | Stability | Use Case |
|---|---|---|---|
| Counting Sort | O(n + k) | Stable | Small integer range (e.g., 0–100) |
| Radix Sort | O(n * d) | Stable | Multi-digit numbers (e.g., phone numbers) |
| Merge Sort | O(n log n) | Stable | General-purpose sorting |

## Advanced Practice

### Conceptual Questions

**Q1:** Why is counting sort more efficient than comparison-based sorts like merge sort when the input range is small?
> Counting sort avoids direct comparisons between elements, achieving linear time complexity O(n + k) instead of O(n log n). When the range of input values (k) is small relative to the number of elements (n), this results in significantly faster sorting.

**Q2:** Why does counting sort require cumulative counts to determine the correct positions of elements?
> Cumulative counts ensure that each element is placed in the correct position by accounting for the number of elements that have already been placed. This prevents overwriting and ensures the sorted order.

**Q3:** Why is the stability of counting sort important in certain applications?
> Stability ensures that elements with equal values retain their relative order from the input. This is crucial in applications like sorting exam scores where maintaining the original order of tied scores might be necessary for fairness or record-keeping.

### Comparison Questions

**Q1:** Under what circumstances should counting sort be preferred over radix sort?
> Counting sort should be preferred over radix sort when the input range (k) is small and the values are integers. Radix sort is more suitable for larger ranges or when sorting multi-digit numbers, as it processes digits one by one.

**Q2:** Under what circumstances should counting sort be preferred over merge sort?
> Counting sort should be preferred over merge sort when the input consists of integers within a small range. Merge sort has a higher time complexity (O(n log n)) compared to counting sort's O(n + k) when the range is small, making counting sort more efficient.

**Q3:** Under what circumstances should counting sort be preferred over radix sort for sorting integers?
> Counting sort should be preferred over radix sort when the range of input values (k) is small relative to the number of elements (n). Radix sort is more efficient for larger ranges or when sorting numbers with multiple digits, as it processes each digit separately.
