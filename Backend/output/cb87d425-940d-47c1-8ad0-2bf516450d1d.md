## File Allocation Methods

**Definition:** File allocation methods are techniques used to manage how files are stored as contiguous or non-contiguous blocks on disk.

File allocation methods determine how disk blocks are allocated for files. Contiguous allocation stores file blocks in contiguous disk locations, offering high performance but suffering from external fragmentation. Linked allocation uses a linked list of blocks, allowing scattered storage but requiring more I/O operations. Indexed allocation employs an index block to map logical to physical blocks, providing flexibility but with higher overhead. Each method has trade-offs in terms of performance, storage efficiency, and management complexity. Contiguous allocation is simple and fast but may require compaction. Linked allocation avoids external fragmentation but can lead to internal fragmentation. Indexed allocation offers efficient block access but requires additional space for the index. These methods impact disk space utilization, file access speed, and system performance.

```cpp
// Example of contiguous allocation in C
#include <stdio.h>
int main() {
    int startBlock = 100;
    int numBlocks = 50;
    printf("File starts at block %d and occupies %d blocks\n", startBlock, numBlocks);
    return 0;
}
```

**Comparison:**
| Allocation Method | Pros | Cons |
|---|---|---|
| Contiguous | Fast access, simple management | External fragmentation, compaction needed |
| Linked | No external fragmentation, flexible storage | Internal fragmentation, slower access |
| Indexed | Efficient block access, no external fragmentation | Higher overhead, requires index block |

> **Example:** A video file stored using contiguous allocation occupies a single, uninterrupted block on the disk, ensuring fast sequential access. In contrast, a text document using linked allocation may have its blocks scattered, requiring the system to traverse pointers to retrieve the entire file.

> [!NOTE]
> **Memory Trick:** Contiguous = Continuous, Linked = Chain of Blocks, Indexed = Index Table.

> [!WARNING]
> **Common Mistake:** Confusing contiguous allocation with linked or indexed methods, misunderstanding the role of the file allocation table in indexed allocation.

> [!TIP]
> **Exam Focus:** Contiguous allocation is the method that stores file blocks in contiguous disk locations. The bit map technique uses a map of bits to track free and occupied blocks.

**Key Takeaways:**
- Contiguous allocation offers fast access but suffers from external fragmentation.
- Linked allocation avoids external fragmentation but may cause internal fragmentation.
- Indexed allocation provides efficient block access with an index block but has higher overhead.



## Advanced Practice

### Conceptual Questions

**Q1:** Why is contiguous allocation more susceptible to external fragmentation compared to indexed allocation?
> Contiguous allocation requires files to occupy contiguous disk blocks, which can lead to external fragmentation when files are deleted and free spaces become scattered. Indexed allocation, on the other hand, uses an index block to map logical to physical blocks, allowing non-contiguous storage and reducing the impact of external fragmentation.

**Q2:** Why does indexed allocation require more disk space compared to contiguous allocation?
> Indexed allocation requires an additional index block to map logical block numbers to physical disk blocks, which consumes extra disk space. Contiguous allocation does not need such an index block, thus using less space but potentially leading to external fragmentation.

**Q3:** Why is linked allocation less efficient for random access compared to indexed allocation?
> Linked allocation stores file blocks in a linked list, requiring sequential traversal to access individual blocks, which is inefficient for random access. Indexed allocation uses an index block to directly map logical block numbers to physical locations, enabling faster random access.

### Comparison Questions

**Q1:** Under what circumstances should contiguous allocation be preferred over indexed allocation?
> Contiguous allocation should be preferred over indexed allocation when the system prioritizes fast sequential access and simple management, and when the file size is relatively small, minimizing the risk of external fragmentation.

**Q2:** Under what circumstances should linked allocation be preferred over indexed allocation?
> Linked allocation should be preferred over indexed allocation in environments with limited disk space, as it does not require an index block. However, this comes at the cost of slower random access and higher I/O operations compared to indexed allocation.

**Q3:** Under what circumstances should indexed allocation be preferred over contiguous allocation?
> Indexed allocation should be preferred over contiguous allocation when the file system needs to support large files, efficient random access, and avoid external fragmentation. This is particularly beneficial in systems where file sizes vary significantly and disk space is adequately available.

### Scenario Questions

**Scenario:** A system experiences frequent external fragmentation due to dynamic file allocation. Which approach do you take to mitigate this issue?
**Expected Answer:**
> Implement contiguous allocation with periodic disk compaction to reduce external fragmentation.

**Scenario:** A system requires efficient random access to file blocks while minimizing disk space overhead. Which allocation method would you recommend?
**Expected Answer:**
> Use indexed allocation as it allows efficient block access without external fragmentation and requires minimal overhead for the index block.

**Scenario:** A system has limited disk space and needs to store a large number of small files efficiently. Which allocation method is most suitable?
**Expected Answer:**
> Opt for linked allocation to avoid external fragmentation and efficiently utilize available disk space for numerous small files.
