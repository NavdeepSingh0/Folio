# 1. Functions of an Operating System

## 📖 What is an Operating System?

An OS is system software that acts as an **interface between user and hardware**, managing resources and providing services.

---

## 📖 Main Functions

| # | Function | Key Activities |
|---|----------|---------------|
| 1 | **Process Management** | Creation, scheduling, termination, synchronization, deadlock handling |
| 2 | **Memory Management** | Allocation/deallocation, tracking usage, virtual memory |
| 3 | **File Management** | File creation/deletion, directory management, access control, backup |
| 4 | **I/O Device Management** | Device drivers, buffering, caching, spooling, allocation |
| 5 | **Resource Allocation** | CPU scheduling, memory allocation, I/O device assignment |
| 6 | **Protection & Security** | Access control, authentication, encryption, malware protection |
| 7 | **User Interface** | CLI, GUI, system calls |
| 8 | **Error Detection & Handling** | Hardware errors, software errors, OS errors (deadlock) |

> **Memory Trick:** "PM-FID-RPS-UE" → **P**rocess, **M**emory, **F**ile, **I**/O, **D**evice, **R**esource, **P**rotection, **S**ecurity, **U**ser Interface, **E**rror Handling

---

## 📖 Diagram: OS as Interface

```
┌──────────────────────────────────────┐
│               USERS                  │
│  (People, Programs, Other Computers) │
└──────────────┬───────────────────────┘
               │
┌──────────────▼───────────────────────┐
│        APPLICATION PROGRAMS          │
│  (Compilers, Browsers, Games, etc.)  │
└──────────────┬───────────────────────┘
               │
┌──────────────▼───────────────────────┐
│          OPERATING SYSTEM            │
│  ┌────────────────────────────────┐  │
│  │ Process │ Memory │ File │ I/O  │  │
│  │ Manager │ Manager│ Mgr  │ Mgr  │  │
│  └────────────────────────────────┘  │
└──────────────┬───────────────────────┘
               │
┌──────────────▼───────────────────────┐
│              HARDWARE                │
│    (CPU, Memory, Disk, I/O Devices)  │
└──────────────────────────────────────┘
```

---

## 📝 Quiz

**Q1.** Which OS function directly prevents two processes from printing simultaneously?
> **Resource Allocation / I/O Device Management** — it allocates the printer to one process at a time.

**Q2.** What is the primary goal of memory management in an OS?
> To track which parts of memory are in use, allocate and deallocate memory to processes, and support virtual memory so processes can run even if they exceed physical RAM.

# 2. Structure of an Operating System

## 📖 Overview

The OS can be internally organized in several ways. Each structure has different trade-offs between performance, modularity, and maintainability.

---

## 📖 1. Monolithic Structure

**Description:** All OS services run in kernel mode as a single large program.

**Examples:** Early UNIX, MS-DOS

```
┌────────────────────────────────────┐
│             User Mode              │
│        (User Applications)         │
└─────────────┬──────────────────────┘
              │ System Calls
┌─────────────▼──────────────────────┐
│            Kernel Mode             │
│  ┌──────────────────────────────┐  │
│  │   All OS Services Together   │  │
│  │  (Process, Memory, File, I/O)│  │
│  └──────────────────────────────┘  │
└─────────────┬──────────────────────┘
              │
┌─────────────▼──────────────────────┐
│             Hardware               │
└────────────────────────────────────┘
```

| Advantages | Disadvantages |
|-----------|---------------|
| Fast execution — no overhead between layers | Difficult to maintain and debug |
| Efficient communication between components | One error can crash entire system |
| | Hard to modify or extend |

---

## 📖 2. Layered Structure

**Description:** OS is divided into layers, each built on top of lower layers.

**Example:** THE Operating System

```
┌────────────────────────────────┐
│   Layer 5: User Programs       │
├────────────────────────────────┤
│   Layer 4: I/O Buffering       │
├────────────────────────────────┤
│   Layer 3: Console Drivers     │
├────────────────────────────────┤
│   Layer 2: Memory Management   │
├────────────────────────────────┤
│   Layer 1: CPU Scheduling      │
├────────────────────────────────┤
│   Layer 0: Hardware            │
└────────────────────────────────┘
```

| Advantages | Disadvantages |
|-----------|---------------|
| Modular — easy to debug and maintain | Slower performance (multiple layer crossings) |
| Each layer can be tested independently | Difficult to define layers properly |
| Clear separation of concerns | |

---

## 📖 3. Microkernel Structure

**Description:** Minimal kernel; most services run in user space.

**Examples:** Mach, QNX, Minix

| Advantages | Disadvantages |
|-----------|---------------|
| More secure and reliable | Performance overhead (frequent user↔kernel switches) |
| Easy to extend and port | More complex communication |
| Isolated services — one crash doesn't affect others | |

---

## 📝 Quiz

**Q3.** Which OS structure would be easiest to debug if a single module fails?
> **Layered Structure** — each layer is isolated and can be tested independently.

**Q4.** Why is the Monolithic structure faster than Microkernel?
> In Monolithic, all services run in kernel mode in one space — no mode switching overhead. In Microkernel, services run in user space and must communicate via kernel, causing frequent context switches.

# 3. Characteristics of Modern Operating Systems

## 📖 Key Characteristics

| # | Characteristic | Description | Example |
|---|---------------|-------------|---------|
| 1 | **Multitasking** | Run multiple tasks; CPU switches rapidly (time-sharing) | Music + browsing simultaneously |
| 2 | **Multiprogramming** | Multiple programs in memory; CPU executes another when one waits for I/O | Maximize CPU utilization |
| 3 | **Multi-user Support** | Multiple users simultaneously, each with isolated environment | UNIX, Linux servers |
| 4 | **Concurrency** | Multiple processes execute "simultaneously"; requires synchronization | Prevents race conditions |
| 5 | **Virtual Memory** | Execute processes larger than physical RAM; uses disk as extended RAM | Each process sees large memory |
| 6 | **Portability** | OS runs on different hardware; written in high-level language (C) | Linux on x86, ARM, RISC-V |
| 7 | **Scalability** | Handles increasing workload; supports multicore systems | Multi-processor systems |
| 8 | **Security & Protection** | Authentication, access control, encryption, process isolation | User login, file permissions |
| 9 | **Reliability & Fault Tolerance** | Error detection/recovery, backup, graceful degradation | RAID, checkpointing |
| 10 | **Efficiency** | Optimal resource utilization, fast response, high throughput | Low idle time |

> **Key Distinction:** Multiprogramming focuses on **CPU utilization**; Multitasking focuses on **user experience (responsiveness)**.

> **Memory Trick:** "MMPV-PSRE" → **M**ultitasking, **M**ultiprogramming, **M**ulti-user, **P**ortability, **V**irtual Memory, **P**rotection, **S**ecurity, **R**eliability, **E**fficiency

---

## 📝 Quiz

**Q5.** A scientific computation runs continuously for 10 minutes — is it CPU-bound or I/O-bound?
> **CPU-bound** — it rarely waits for I/O, spending most time using the processor.

# 4. Types of Operating Systems

## 📖 1. Batch Operating System

Jobs are collected in batches and processed sequentially with **no user interaction** during execution.

**Examples:** Payroll processing, bank statements, IBM OS/360

| Advantages | Disadvantages |
|-----------|---------------|
| Efficient for large repetitive jobs | No interaction during execution |
| Reduced idle time | Difficult to debug |
| | Priority cannot be changed mid-run |

---

## 📖 2. Multiprogramming OS

Multiple programs loaded in memory; CPU switches when one waits for I/O. **Goal: keep CPU always busy.**

```
Memory:
┌─────────┐
│   OS    │
├─────────┤
│ Job 1   │ ← Currently using CPU
├─────────┤
│ Job 2   │ ← Waiting for I/O
├─────────┤
│ Job 3   │ ← Ready to execute
└─────────┘
```

---

## 📖 3. Time-Sharing / Multitasking OS

CPU time divided into small time slices (quantum). Each process gets CPU for one quantum.

**Examples:** Windows, Linux, macOS

**Time Quantum:** Typically 10–100 milliseconds

| Advantages |
|-----------|
| Good response time |
| Multiple users can work simultaneously |
| Better CPU utilization |

---

## 📖 4. Real-Time Operating System (RTOS)

| Type | Deadline | Consequence of Miss | Examples |
|------|----------|---------------------|---------|
| **Hard Real-Time** | Must be met | System failure | Air traffic control, medical devices, missiles |
| **Soft Real-Time** | Should be met | Degraded performance acceptable | Video streaming, online gaming, multimedia |

---

## 📖 5. Distributed Operating System

Multiple independent computers appear as a single system; resources shared across network.

**Example:** Cluster computing

**Features:** Resource sharing, computation speedup, reliability (redundancy), communication

---

## 📖 6. Network Operating System

Provides networking capabilities; each computer has its own OS but can share resources.

**Examples:** Windows Server, Novell NetWare

> **Difference from Distributed:** In NOS, users are **aware** they are on a network. In Distributed OS, it is **transparent** to users.

---

## 📖 7. Embedded OS

Designed for specific devices with limited resources (memory, power).

**Device Examples:** Car systems, washing machines, routers

**OS Examples:** Embedded Linux, Windows CE, VxWorks, FreeRTOS

---

## 📖 8. Mobile OS

For mobile devices (phones, tablets). Touch-based interface, power-efficient.

**Examples:** Android, iOS

---

## 📖 Comparison Table

| Type | Example | Key Feature | Use Case |
|------|---------|-------------|---------|
| Batch | IBM OS/360 | No interaction | Payroll, billing |
| Time-Sharing | UNIX, Windows | Multiple users | General computing |
| Real-Time | QNX, VxWorks | Deadline-driven | Industrial control |
| Distributed | Amoeba | Multi-computer | Cluster computing |
| Network | Windows Server | Resource sharing | File/print servers |
| Embedded | FreeRTOS | Resource-constrained | IoT devices |
| Mobile | Android, iOS | Touch, power-efficient | Smartphones |

---

## 📝 Quiz

**Q6.** A pacemaker controlling a heartbeat — which type of OS would it use and why?
> **Hard Real-Time OS** — failure to meet a timing deadline is life-threatening and constitutes a system failure.

**Q7.** What is the difference between a Network OS and a Distributed OS?
> In a Network OS, users know they are accessing remote resources (not transparent). In a Distributed OS, the system appears as a single unified machine — network details are hidden from users.

# 5. System Calls – Role and Working

## 📖 What is a System Call?

A **programming interface** that allows user programs to request services from the OS kernel.

**Why needed:** User programs run in **user mode** (restricted). The kernel runs in **kernel mode** (full hardware access). System calls are the safe bridge between them.

---

## 📖 How System Calls Work

```
User Program
    │
    │  1. Call library function (e.g., printf())
    ▼
C Library (wrapper)
    │
    │  2. Prepare system call
    ▼
Trap / Interrupt
    │
    │  3. Switch to kernel mode
    ▼
Kernel
    │  4. Execute system call
    │  5. Perform requested operation
    ▼
Return to User Mode
    │
    │  6. Return result
    ▼
User Program
```

---

## 📖 Example: Reading a File

```c
FILE *fp = fopen("test.txt", "r");   // System call: open()
char buffer[100];
fread(buffer, 1, 100, fp);           // System call: read()
fclose(fp);                          // System call: close()
```

**Behind the scenes:**
1. `fopen()` → calls `open()` system call
2. OS checks permissions
3. OS allocates file descriptor
4. Returns handle to user program

---

## 📖 Role of System Calls

| Role | Description |
|------|-------------|
| **Interface** | Safe way to access kernel services; prevents direct hardware access |
| **Resource Management** | Memory (`malloc` → `brk`), files, processes |
| **Protection** | Validates parameters, enforces access control |
| **Abstraction** | Hides hardware complexity; uniform interface across hardware |

---

## 📝 Quiz

**Q8.** Why can't a user program directly access hardware?
> User programs run in restricted **user mode** to prevent unauthorized or accidental hardware access. Only the kernel (running in **kernel mode**) can directly manipulate hardware. System calls are the controlled mechanism to request these services.

# 6. Types of System Calls

## 📖 Category Overview

### 1. Process Control

**Purpose:** Create, terminate, control processes

| System Call | Purpose |
|-------------|---------|
| `fork()` | Create child process |
| `exec()` | Load new program |
| `exit()` | Terminate process |
| `wait()` | Wait for child process |
| `getpid()` | Get process ID |
| `kill()` | Send signal to process |

```c
int pid = fork();          // Create child process
if (pid == 0) {
    exec("/bin/ls");       // Child: execute 'ls'
    exit(0);
} else {
    wait(NULL);            // Parent: wait for child
}
```

---

### 2. File Management

**Purpose:** Create, read, write, delete files

| System Call | Purpose |
|-------------|---------|
| `open()` | Open file |
| `read()` | Read from file |
| `write()` | Write to file |
| `close()` | Close file |
| `lseek()` | Move file pointer (random access) |
| `stat()` | Get file info (size, permissions) |
| `unlink()` | Delete file |

```c
int fd = open("data.txt", O_RDONLY);
char buffer[1024];
read(fd, buffer, 1024);
close(fd);
```

---

### 3. Device Management

**Purpose:** Request/release devices, read/write

| System Call | Purpose |
|-------------|---------|
| `ioctl()` | Device-specific operations |
| `read()` | Read from device (e.g., keyboard) |
| `write()` | Write to device (e.g., screen) |

---

### 4. Information Maintenance

**Purpose:** Get/set system information

| System Call | Purpose |
|-------------|---------|
| `getpid()` | Get process ID |
| `time()` | Get current time |
| `getuid()` | Get user ID |
| `sysinfo()` | Get system info (memory, uptime) |

---

### 5. Communication

**Purpose:** Inter-process communication

| System Call | Purpose |
|-------------|---------|
| `pipe()` | Create pipe |
| `shmget()` | Create shared memory |
| `msgget()` | Create message queue |
| `socket()` | Create network socket |
| `send()` / `recv()` | Send/receive data over socket |

---

### 6. Protection

**Purpose:** Control access to resources

| System Call | Purpose |
|-------------|---------|
| `chmod()` | Change file permissions |
| `chown()` | Change file owner |
| `umask()` | Set default permissions for new files |

---

## 📝 Quiz

**Q9.** Which system call category would a web server use most heavily?
> **Communication** (sockets for client connections) and **File Management** (serving files). It would also use **Process Control** (forking to handle concurrent clients).

# 7. System Programs – Categories and Functions

## 📖 What are System Programs?

**Utilities** that provide a convenient environment for program development and execution. They sit between user applications and the OS kernel.

---

## 📖 System Calls vs System Programs

| Aspect | System Calls | System Programs |
|--------|-------------|-----------------|
| What | Low-level API (kernel entry points) | High-level utilities |
| Who uses | Programmers | End users & programmers |
| Example | `read()`, `write()` | `ls`, `cp`, `grep` |
| Mode | Kernel mode entry | Use system calls internally |

---

## 📖 Categories

### 1. File Management
**Programs:** `cp`, `mv`, `rm`, `ls`, `cat`, `mkdir`

```bash
ls -l          # List files with details
cp file1 file2 # Copy file
mkdir folder   # Create directory
```

### 2. Status Information
**Programs:** `date`, `ps`, `top`, `df`, `free`

```bash
ps aux         # List all running processes

df -h          # Disk space usage
```

### 3. File Modification
**Programs:** `vi`, `nano`, `grep`, `sed`, `awk`

```bash
grep "error" log.txt        # Search in file
sed 's/old/new/g' file      # Find and replace
```

### 4. Programming Language Support
**Programs:** `gcc`, `g++`, `python`, `java`, `gdb`

```bash
gcc program.c -o program    # Compile C program
gdb program                 # Debug program
```

### 5. Program Loading and Execution
**Programs:** `ld` (linker), loaders

- Load executable into memory
- Resolve external references
- Start program execution

### 6. Communications
**Programs:** `ssh`, `ftp`, `scp`, `mail`

```bash
ssh user@remote.com              # Remote login

```

### 7. Background Services (Daemons)
**Programs:** `cron`, `syslog`, `httpd`, `sshd`

- `cron` — Schedule tasks

- `httpd` — Web server daemon

# 8. Process Concept and Process States

## 📖 What is a Process?

A **program in execution**. It includes:

| Component | Description |
|-----------|-------------|
| Program code (text section) | Instructions |
| Program counter & registers | Current execution point |
| Stack | Temporary data — function parameters, return addresses, local variables |
| Data section | Global variables |
| Heap | Dynamically allocated memory |

> **Program vs Process:** A program is a **passive entity** (file on disk). A process is an **active entity** (program being executed).
> Analogy: Recipe (Program) vs Cooking (Process)

---

## 📖 Five Process States

| State | Description |
|-------|-------------|
| **New** | Process is being created; resources being allocated |
| **Ready** | Waiting for CPU; has all resources except CPU; sits in ready queue |
| **Running** | Instructions being executed; process has the CPU |
| **Waiting / Blocked** | Waiting for an event (I/O completion, signal); cannot proceed even if CPU is free |
| **Terminated** | Finished execution; resources being deallocated |

> **Key Rule:** Only **ONE** process can be in Running state at a time (single CPU). Multiple processes can be in Ready or Waiting simultaneously.

---

## 📖 Process State Transition Diagram

```
         ┌─────┐
         │ NEW │
         └──┬──┘
            │ admitted
            ▼
         ┌──────┐
    ┌───▶│READY │◀──────────────────────────┐
    │    └──┬───┘                           │
    │       │ scheduler dispatch            │ I/O or event completion
    │       ▼                               │
    │    ┌─────────┐    interrupt     ──────┘
    │    │ RUNNING │────────────────►(Ready)
    │    └─────┬───┘
    │           │
    │           │ I/O or event wait
    │           ▼
    │    ┌───────────┐
    │    │ WAITING / │
    │    │  BLOCKED  │
    │    └─────┬─────┘
    │           │ I/O or event completion
    └───────────┘
         │ exit
         ▼
    ┌──────────┐
    │TERMINATED│
    └──────────┘
```

---

## 📖 State Transitions Explained

| Transition | Cause |
|-----------|-------|
| New → Ready | OS admits process into memory; PCB assigned |
| Ready → Running | CPU scheduler selects the process |
| Running → Ready | Time quantum expired; higher-priority process arrived (preemption) |
| Running → Waiting | Process requests I/O; waits for user input or resource |
| Waiting → Ready | I/O completed; event occurred; resource available |
| Running → Terminated | Process completes; killed; error occurred |

---

## 📖 Scheduling Queues

```
┌──────────────┐
│  Job Queue   │  ← All processes in system
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ Ready Queue  │  ← In memory, ready to execute
└──────┬───────┘
       │
       ▼
     ┌─────┐
     │ CPU │
     └──┬──┘
        │
        ├────────────────────┐
        │                    ▼
        │             ┌─────────────┐
        │             │Device Queues│  ← Waiting for I/O
        │             └──────┬──────┘
        └────────────────────┘
               (back to ready queue after I/O)
```

---

## 📝 Quiz

**Q11.** Can a process move directly from Waiting to Running? Why or why not?
> **No.** When I/O completes, the process moves from Waiting → **Ready**. It must then wait for the CPU scheduler to select it before moving to Running.

# 9. Process Control Block (PCB)

## 📖 What is a PCB?

A **data structure** maintained by the OS for every process. Contains all information needed to manage and resume that process.

**Also called:** Task Control Block (TCB), Process Descriptor

**Why needed:** When the CPU switches between processes, it must save and restore each process's complete state — the PCB holds that state.

---

## 📖 Components of PCB

```
┌─────────────────────────────────────┐
│         PROCESS CONTROL BLOCK       │
├─────────────────────────────────────┤
│ 1. Process ID (PID)                 │
├─────────────────────────────────────┤
│ 2. Process State                    │
│    (New / Ready / Running /         │
│     Waiting / Terminated)           │
├─────────────────────────────────────┤
│ 3. Program Counter (PC)             │
│    (Address of next instruction)    │
├─────────────────────────────────────┤
│ 4. CPU Registers                    │
│    (Accumulators, Index regs,       │
│     Stack pointer, GPRs)            │
├─────────────────────────────────────┤
│ 5. CPU Scheduling Information       │
│    - Priority                       │
│    - Queue pointers                 │
│    - Time quantum / Burst time      │
├─────────────────────────────────────┤
│ 6. Memory Management Information    │
│    - Base & limit registers         │
│    - Page tables / Segment tables   │
├─────────────────────────────────────┤
│ 7. Accounting Information           │
│    - CPU time used                  │
│    - Time limits / Account number   │
├─────────────────────────────────────┤
│ 8. I/O Status Information           │
│    - Open files list                │
│    - Allocated I/O devices          │
│    - Pending I/O requests           │
├─────────────────────────────────────┤
│ 9. Parent Process Pointer           │
├─────────────────────────────────────┤
│ 10. Child Process Pointers          │
└─────────────────────────────────────┘
```

---

## 📖 Context Switching

When the CPU switches from one process to another:

```
Process P0 Running
    │
    │ 1. Interrupt / system call
    ▼
Save P0's state in PCB0
    (program counter, registers, state = Ready)
    │
    ▼
Load P1's state from PCB1
    (restore program counter, registers, state = Running)
    │
    ▼
Execute P1
```

> **Overhead:** Context switching is **pure overhead** — the system does no useful work during the switch. Typically takes 1–10 microseconds.

---

## 📝 Quiz

**Q12.** Why is the Program Counter the most critical field in a PCB?
> The Program Counter stores the **address of the next instruction** to execute. Without it, the OS cannot resume a process from exactly where it was paused during a context switch.

# 10. Program vs Process

## 📖 Comprehensive Comparison

| Aspect | Program | Process |
|--------|---------|---------|
| **Definition** | Passive entity stored on disk | Active entity (program in execution) |
| **Nature** | Static | Dynamic |
| **Lifespan** | Permanent (until deleted) | Temporary (while running) |
| **Resources** | None — just disk space | Uses CPU, memory, I/O |
| **PCB** | No PCB | Has PCB |
| **State** | No state | Has state (New/Ready/Running/etc.) |
| **Contents** | Code + static data | Code + data + stack + heap + PCB |
| **Count** | One file | Multiple processes from one program |
| **Example** | `notepad.exe` file on disk | Running instance of Notepad |

---

## 📖 Key Example

```
Program: Microsoft Word (word.exe)
              ↓
    User opens 3 documents
              ↓
    Creates 3 Processes:
    ┌─────────────────────────────┐
    │ Process 1 (PID: 1001)       │ ← Document1.docx
    │ Process 2 (PID: 1002)       │ ← Document2.docx
    │ Process 3 (PID: 1003)       │ ← Document3.docx
    └─────────────────────────────┘
    Each has: own memory, own PCB, independent execution
```

---

## 📝 Quiz

**Q13.** One program — three instances running simultaneously. Are they the same process?
> **No.** Each running instance is a separate **process** with its own PID, PCB, memory space, and state. The program (the file on disk) is shared, but each execution creates an independent process.

# 11. Process Scheduling

## 📖 What is Process Scheduling?

The method by which the OS decides **which process gets the CPU** and **when**.

**Why needed:** Multiprogramming, fairness, efficiency, response time.

---

## 📖 Three Types of Schedulers

```
┌──────────────────────────────────────┐
│     LONG-TERM SCHEDULER (Job)        │
│ Selects jobs from pool → memory      │
│ Runs infrequently (seconds/minutes)  │
└────────────┬─────────────────────────┘
             │
             ▼ Job Queue
┌──────────────────────────────────────┐
│   SHORT-TERM SCHEDULER (CPU) ⭐       │
│ Selects from ready queue → CPU       │
│ Runs very frequently (milliseconds)  │
└────────────┬─────────────────────────┘
             │
             ▼ CPU
┌──────────────────────────────────────┐
│     MEDIUM-TERM SCHEDULER            │
│ Swapping — removes/adds processes    │
│ Runs when memory is short            │
└──────────────────────────────────────┘
```

---

## 📖 Scheduler Details

### Long-Term (Job Scheduler)
- Controls **degree of multiprogramming** (how many processes in memory)
- Goal: Good mix of **I/O-bound** and **CPU-bound** processes
  - **I/O-bound:** Spends more time doing I/O (e.g., text editor)
  - **CPU-bound:** Spends more time on CPU (e.g., scientific computation)

### Short-Term (CPU Scheduler)
- Selects from **ready queue** and gives CPU
- Must be **FAST** (runs 100+ times per second)
- Most important scheduler — affects system responsiveness directly

### Medium-Term (Swapper)
- **Swap out:** Remove process from memory to disk when memory is short
- **Swap in:** Bring process back to memory when needed
- Controls degree of multiprogramming dynamically

---

## 📖 Context Switch

Steps when CPU switches processes:
1. Save state of current process in PCB (PC, registers, memory info)
2. Scheduler selects next process
3. Load state of next process from its PCB
4. Resume execution

**Factors affecting switch time:** Number of registers, memory speed, special hardware support.

---

## 📖 Role of CPU Scheduler — Goals

| Goal | Metric | Optimize |
|------|--------|---------|
| CPU Utilization | % time CPU is busy | Maximize |
| Throughput | Processes completed per time unit | Maximize |
| Turnaround Time | Submission → Completion | Minimize |
| Waiting Time | Time in ready queue | Minimize |
| Response Time | Submission → First response | Minimize |
| Fairness | Equal CPU share | Ensure no starvation |

---

## 📝 Quiz

**Q14.** Why must the short-term scheduler be extremely fast?
> It runs **hundreds of times per second** (every 10–100 ms for Round Robin). If scheduling itself took significant time, a large fraction of CPU would be wasted on overhead rather than running user processes.

# 12. Threads – Types and Advantages

## 📖 What is a Thread?

A **lightweight process** — the basic unit of CPU utilization within a process.

**A thread has its own:**
- Thread ID, Program Counter, Register set, Stack

**Threads share (within the same process):**
- Code section, Data section, Heap, OS resources (open files, signals)

```
Single-Threaded Process:      Multi-Threaded Process:
┌──────────────────┐          ┌──────────────────┐
│      Code        │          │      Code        │  ← Shared
├──────────────────┤          ├──────────────────┤
│      Data        │          │      Data        │  ← Shared
├──────────────────┤          ├──────────────────┤
│      Heap        │          │      Heap        │  ← Shared
├──────────────────┤          ├──────┬──────┬────┤
│     Stack (T1)   │          │Stack │Stack │Stack  ← Separate
└──────────────────┘          │(T1)  │(T2)  │(T3)│
                              └──────┴──────┴────┘
```

---

## 📖 Types of Threads

### 1. User-Level Threads (ULT)

Managed by a **user-level thread library**. Kernel is **unaware** of these threads.

**Libraries:** POSIX Pthreads, Win32 threads, Java threads

| Advantages | Disadvantages |
|-----------|---------------|
| Fast context switching (no kernel mode switch) | Blocking system call blocks the **entire process** |
| Portable — works on any OS | Cannot use multiple CPUs (kernel sees one process) |
| Flexible, custom scheduling possible | No system-wide coordination |

**Key Problem:**
```
T1 makes blocking system call
→ Kernel blocks entire process P1
→ T2, T3 also blocked (even though they're ready!)
```

### 2. Kernel-Level Threads (KLT)

Managed **directly by the OS kernel**. Kernel is aware of every thread.

**Examples:** Windows threads, Linux `clone()`, Solaris threads

| Advantages | Disadvantages |
|-----------|---------------|
| True parallelism on multiprocessors | Slower — context switch requires kernel mode |
| Non-blocking: if one thread blocks, others continue | Limited number (kernel resources needed) |
| Independent kernel scheduling per thread | Less portable (OS-specific) |

### 3. Hybrid (Combined) Threads

Many user threads mapped to fewer kernel threads (M:N model).

**Example:** Solaris — developer controls the mapping.

**Benefit:** Balance between ULT flexibility and KLT parallelism.

---

## 📖 Comparison Table

| Feature | User-Level | Kernel-Level |
|---------|-----------|--------------|
| Management | Thread library | OS kernel |
| Context switch | Fast | Slow |
| Blocking call | Blocks all threads | Only blocking thread |
| Multiprocessor use | No | Yes |
| Overhead | Low | High |
| Example | Pthreads | Windows threads |

---

## 📖 Advantages of Threads Over Processes

### 1. Responsiveness
Multi-threaded app continues even if part is blocked.

**Example — Web Browser:**
- Thread 1: Displays images
- Thread 2: Fetches data
- User can scroll while loading

### 2. Resource Sharing
Threads share memory — no IPC needed for communication.

```
4 Processes (wasteful):         4 Threads (efficient):
┌────┐ ┌────┐ ┌────┐ ┌────┐    ┌─────────────────┐
│Mem │ │Mem │ │Mem │ │Mem │    │  Process        │
└────┘ └────┘ └────┘ └────┘    │  Shared Mem+Code│
                                │  T1 │T2 │T3 │T4 │
                                └─────┴───┴───┴───┘
```

### 3. Economy
- Thread creation: ~30× faster than process creation
- Thread context switch: ~5× faster

### 4. Scalability (Multiprocessor)
Threads run in **parallel** on multiple CPU cores.

### 5. Simplified Code Structure

**Example — Video Player:**
```
Thread 1: Video decoding
Thread 2: Audio decoding
Thread 3: UI updates
Thread 4: Network buffering
```

---

## 📝 Quiz

**Q15.** Why do User-Level Threads fail to exploit multi-core CPUs?
> The kernel sees the entire process as a **single entity**. Even if the process has multiple ULTs, the kernel schedules it as one unit and assigns it to one core. KLTs allow the kernel to schedule individual threads across multiple cores.

# 13. Process vs Thread

## 📖 Comprehensive Comparison

| Aspect | Process | Thread |
|--------|---------|--------|
| **Definition** | Independent program in execution | Lightweight unit within a process |
| **Resource ownership** | Own memory space | Shares process memory |
| **Communication** | IPC (pipes, sockets) — complex | Shared memory — simple |
| **Creation time** | Slow (~10–100 microseconds) | Fast (~1–10 microseconds) |
| **Context switch** | Expensive | Cheap |
| **Isolation** | Fully isolated | Share address space |
| **Protection** | Protected from each other | Minimal protection |
| **Overhead** | High | Low |
| **Blocking** | Independent | One thread blocking can affect others |

---

## 📖 Memory Layout Comparison

```
TWO PROCESSES:                   ONE PROCESS WITH TWO THREADS:

Process P1:    Process P2:       Process P:
┌──────────┐  ┌──────────┐       ┌───────────────────┐

├──────────┤  ├──────────┤       ├───────────────────┤
│  Data    │  │  Data    │       │      Data         │ ← Shared
├──────────┤  ├──────────┤       ├───────────────────┤
│  Stack   │  │  Stack   │       │  Stack   │ Stack  │ ← Separate

└──────────┘  └──────────┘       │  Heap (shared)    │

```

## 📖 Quick Reference

| | Processes | Threads |
|--|-----------|---------|
| Memory | 🔒 Isolated | 🔓 Shared |
| Speed | 🐌 Heavy | ⚡ Light |
| Safety | 🛡️ One crash doesn't affect others | ⚠️ One crash can affect all |
| Communication | 📞 IPC needed | 💬 Direct shared memory |

---

## 📝 Quiz

**Q16.** A word processor has spell-check running in the background. Should it use a separate process or thread?
> **Thread** — spell-check needs to access the same document data that the main editor uses. Using threads lets both share memory directly (no IPC needed), and it's faster and more efficient. A separate process would require complex IPC just to share the document content.

# 14. Preemptive vs Non-Preemptive Scheduling

## 📖 Non-Preemptive Scheduling

Once CPU is allocated, the process **keeps it until:**
- It terminates, OR
- It voluntarily releases CPU (I/O wait)

**Scheduling decision points:** Process terminates, or Running → Waiting.

| Advantages | Disadvantages |
|-----------|---------------|
| Simple to implement | Poor response time for interactive systems |
| Low overhead (no preemption) | **Convoy effect** — short processes wait for long ones |
| No synchronization issues | Starvation possible |

**Examples:** FCFS, SJF (non-preemptive), Priority (non-preemptive)

---

## 📖 Preemptive Scheduling

CPU can be **forcibly taken** from a running process before it finishes.

**When preemption occurs:**
- Higher-priority process arrives
- Time quantum expires
- Interrupt occurs

**Scheduling decision points:** All 4 — terminate, Running→Waiting, Running→Ready, Waiting→Ready.

| Advantages | Disadvantages |
|-----------|---------------|
| Better response time | More complex to implement |
| Fair CPU distribution | Higher overhead (frequent context switches) |
| Prevents long processes from hogging CPU | Synchronization needed (race conditions possible) |
| Suitable for time-sharing | Shared data issues |

**Examples:** Round Robin, SRTF, Priority (preemptive)

---

## 📖 Comparison Table

| Feature | Non-Preemptive | Preemptive |
|---------|----------------|------------|
| CPU Release | Voluntary | Forced by OS |
| Response Time | Poor | Good |
| Complexity | Simple | Complex |
| Overhead | Low | High |
| Fairness | Poor | Good |
| Starvation | Possible | Less likely |
| Use Case | Batch systems | Time-sharing, Real-time |

---

## 📖 Illustrated Example

**Non-Preemptive (FCFS):**
```
P1: AT=0, BT=20    P2: AT=1, BT=3    P3: AT=2, BT=2

0              20  23  25
|─────── P1 ───────|P2|P3|

P2 waited 19 units — poor response!
```

**Preemptive (Round Robin, quantum=4):**
```
0   4   8  12  16  19  22  25
|P1|P2|P3|P1 |P1 |P1 |P1 |P2|

P2 starts at t=4 — waited only 3 units!
```

---

## 📝 Quiz

**Q17.** A hospital OS manages both life-support monitors (critical) and billing software (non-critical). Which type of scheduling is essential?
> **Preemptive** — the life-support process must be able to **preempt** the billing software immediately when a critical event occurs. Non-preemptive would let billing run to completion, potentially delaying critical care.

# 15. CPU Scheduling Criteria

## 📖 Performance Metrics

### 1. CPU Utilization

```
CPU Utilization = (Total CPU busy time / Total time) × 100%
```

**Goal:** Maximize | **Range:** 40% (light load) to 90% (heavy load)

### 2. Throughput

```
Throughput = Number of processes completed / Total time
```

**Goal:** Maximize | **Units:** processes/second

### 3. Turnaround Time (TAT)

```
Turnaround Time = Completion Time − Arrival Time
               = Waiting Time + Burst Time
```

**Goal:** Minimize

### 4. Waiting Time (WT)

```
Waiting Time = Turnaround Time − Burst Time
```

**Goal:** Minimize

### 5. Response Time (RT)

```
Response Time = First CPU Time − Arrival Time
```

**Goal:** Minimize — especially critical for **interactive** systems

> **Key Distinction:** Response Time = time to **START**; Turnaround Time = time to **COMPLETE**.

### 6. Fairness

Each process gets a **fair share** of CPU. No starvation. Measured by variance in waiting times.

---

## 📖 Which Algorithm Optimizes What?

| Algorithm | Primary Optimization |
|-----------|---------------------|
| FCFS | Simplicity |
| SJF | Minimum average waiting time |
| SRTF | Minimum average turnaround time |
| Priority | Important processes first |
| Round Robin | Response time and fairness |

---

## 📖 Example Calculation (FCFS)

```
Process | AT | BT
  P1    |  0 |  5
  P2    |  1 |  3
  P3    |  2 |  2

Gantt:   0      5      8  10
         |─ P1 ─|─ P2 ─|P3|

P  | AT | BT | CT | TAT | WT
P1 |  0 |  5 |  5 |  5  |  0
P2 |  1 |  3 |  8 |  7  |  4
P3 |  2 |  2 | 10 |  8  |  6

Average TAT = (5+7+8)/3 = 6.67
Average WT  = (0+4+6)/3 = 3.33
```

---

## 📝 Quiz

**Q18.** Process P has AT=2, BT=6, CT=12. Calculate TAT and WT.
> TAT = CT − AT = 12 − 2 = **10**
> WT = TAT − BT = 10 − 6 = **4**

# 16. CPU Scheduling Algorithms

## 📖 1. First-Come, First-Served (FCFS)

**Type:** Non-preemptive | **Structure:** FIFO queue

**Logic:** Process that arrives first gets CPU first.

| Advantages | Disadvantages |
|-----------|---------------|
| Simple to implement | **Convoy effect** — short processes wait behind long ones |
| Fair ordering | Poor average waiting time |

```
P1: AT=0, BT=24    P2: AT=1, BT=3    P3: AT=2, BT=3

Gantt:  0              24   27   30
        |──────── P1 ────────|P2 |P3 |

WT:  P1=0,  P2=23,  P3=25
Avg WT = (0+23+25)/3 = 16
```

---

## 📖 2. Shortest Job First (SJF)

**Type:** Non-preemptive | **Proven optimal** for average waiting time.

**Logic:** Process with the **smallest burst time** runs first.

| Advantages | Disadvantages |
|-----------|---------------|
| Optimal average waiting time | **Starvation** — long processes may never run |
| Reduces waiting time | Burst time is difficult/impossible to know in advance |

```
All arrive at t=0:  P1(BT=6), P2(BT=8), P3(BT=7), P4(BT=3)

Order: P4 → P1 → P3 → P2

Gantt:  0   3     9       16          24
        |P4 |─P1─|─ P3 ──|─── P2 ─────|

WT:  P4=0, P1=3, P3=9, P2=16
Avg WT = (0+3+9+16)/4 = 7
```

---

## 📖 3. Shortest Remaining Time First (SRTF)

**Type:** Preemptive version of SJF.

**Logic:** At every scheduling point, run the process with the **smallest remaining burst time**.

| Advantages | Disadvantages |
|-----------|---------------|
| Minimum average waiting time | Starvation of longer processes |
| Better response than SJF | More context switches |

```
P1(AT=0, BT=8)  P2(AT=1, BT=4)  P3(AT=2, BT=9)  P4(AT=3, BT=5)

t=0: P1 starts (rem=8)
t=1: P2 arrives (rem=4 < 7) → Preempt P1, run P2
t=5: P2 completes → Run P4 (rem=5 < 7 < 9)
t=10: P4 completes → Run P1 (rem=7 < 9)
t=17: P1 completes → Run P3
t=26: P3 completes

Gantt: 0 1    5     10       17         26
       |P1|P2 ─|─P4──|───P1───|──── P3 ─|
```

---

## 📖 4. Priority Scheduling

**Type:** Preemptive or Non-preemptive.

**Logic:** Each process has a **priority number** (lower = higher priority). Highest priority process gets CPU.

**Starvation problem → Solution: Aging** (gradually increase priority of waiting processes)

```
P1(AT=0, BT=4, Pri=2)  P2(AT=1, BT=3, Pri=1)  P3(AT=2, BT=1, Pri=3)  P4(AT=3, BT=5, Pri=4)

Non-Preemptive:
Gantt: 0    4   7  8       13
       |─P1─|P2 |P3|──P4──|

Execution order: P1 (arrives first) → P2 (priority 1) → P3 → P4
```

---

## 📖 5. Round Robin (RR) ⭐ Most Important for Exam

**Type:** Preemptive.

**Logic:** Each process gets a fixed **time quantum (q)**. After quantum expires, process moves to the end of the ready queue.

| Advantages | Disadvantages |
|-----------|---------------|
| **Fair** — all processes get equal time | Average waiting time can be high |
| **No starvation** | Performance depends critically on quantum size |
| **Good response time** | More context switches |
| Best for **time-sharing** | No priority differentiation |

**Effect of Time Quantum:**

```
q too LARGE → Becomes like FCFS (poor response)
q too SMALL → Too many context switches (high overhead)
OPTIMAL q   → Slightly larger than typical CPU burst (10–100 ms)
```

---

## 📖 Round Robin — Detailed Example

```
P1(AT=0, BT=5)  P2(AT=1, BT=3)  P3(AT=2, BT=1)  P4(AT=3, BT=2)
Time Quantum = 2

t=0–2:  P1 runs (rem=3).  Queue: [P2]
t=2–4:  P2 runs (rem=1).  Queue: [P3, P1]
t=4–5:  P3 runs, completes. Queue: [P1, P4]
t=5–7:  P1 runs (rem=1).  Queue: [P4, P2]
t=7–9:  P4 runs, completes. Queue: [P2, P1]
t=9–10: P2 runs, completes. Queue: [P1]
t=10–11: P1 runs, completes.

Gantt: 0  2  4 5  7  9 10 11
       |P1|P2|P3|P1|P4|P2|P1|

CT: P1=11, P2=10, P3=5, P4=9

TAT: P1=11-0=11, P2=10-1=9, P3=5-2=3, P4=9-3=6
WT:  P1=11-5=6,  P2=9-3=6,  P3=3-1=2, P4=6-2=4

Avg TAT = (11+9+3+6)/4 = 7.25
Avg WT  = (6+6+2+4)/4  = 4.5
```

---

## 📖 Algorithm Comparison

| Algorithm | Type | Avg WT | Starvation | Best Use Case |
|-----------|------|--------|------------|---------------|
| FCFS | Non-pre | Poor | No | Simple batch |
| SJF | Non-pre | **Best** | Yes (long) | Known burst times |
| SRTF | Preemptive | Excellent | Yes | Predictable workloads |
| Priority | Both | Variable | Yes | Important tasks first |
| RR | Preemptive | Moderate | **No** | Time-sharing |

---

## 📝 Quiz

**Q19.** With RR and quantum=4, three processes P1(BT=12), P2(BT=4), P3(BT=6) all arrive at t=0. Which completes first?
> P2 completes at t=8 (gets quantum 0–4, then 4–8 to finish). P3 completes at t=18. P1 completes at t=22. **P2 finishes first.**

# 17. Inter-Process Communication (IPC)

## 📖 What is IPC?

A mechanism that allows processes to **communicate** and **synchronize** their actions.

**Why IPC is needed:**
- Information sharing between processes
- Computation speedup (divide task)
- Modularity (separate processes for separate roles)
- Convenience (user working on multiple tasks)

**Types of Processes:**
- **Independent:** Cannot affect/be affected by others; does not share data
- **Cooperating:** Shares data; can affect others (e.g., Producer-Consumer)

---

## 📖 IPC Mechanisms Overview

```
┌─────────────────────────────────────┐
│           IPC MECHANISMS            │
├─────────────────────────────────────┤
│  1. Shared Memory                   │
│  2. Message Passing                 │
│  3. Pipes (Ordinary & Named)        │
│  4. Message Queues                  │
│  5. Sockets                         │
│  6. Signals                         │
└─────────────────────────────────────┘
```

---

## 📖 1. Shared Memory

Processes share a **common memory region**. OS does NOT intervene during data exchange after setup.

```
Process P1       Shared Memory       Process P2
┌────────┐      ┌─────────────┐      ┌────────┐
│  Code  │◄────►│   Shared    │◄────►│  Code  │
│  Data  │ R/W  │   Region    │ R/W  │  Data  │
└────────┘      └─────────────┘      └────────┘
```

```c
// Producer:
int shmid = shmget(key, size, IPC_CREAT | 0666);
char *ptr  = shmat(shmid, NULL, 0);
strcpy(ptr, "Hello from P1");

// Consumer:
int shmid = shmget(key, size, 0666);
char *ptr  = shmat(shmid, NULL, 0);
printf("%s", ptr);
```

| Advantages | Disadvantages |
|-----------|---------------|
| **Fast** — no kernel intervention after setup | **Synchronization needed** — race conditions |
| Direct memory access | Complex — programmer handles sync |
| Efficient for large data | No protection — process can corrupt data |

---

## 📖 2. Message Passing

Processes send/receive **messages via the OS**.

```
Process P1         OS Kernel         Process P2
┌────────┐      ┌─────────────┐      ┌────────┐
│ send() ├─────►│   Message   │◄─────┤ recv() │
│        │      │    Queue    │      │        │
└────────┘      └─────────────┘      └────────┘
```

**Communication types:**
- **Direct:** `send(P2, message)` — explicitly name the other process
- **Indirect:** `send(mailbox_A, message)` — via mailbox/port

| Advantages | Disadvantages |
|-----------|---------------|
| No synchronization issues — OS handles | **Slower** — kernel involved in every operation |
| Protection — OS controls access | Overhead from copying messages |
| Works on distributed systems | Message size restrictions |

---

## 📖 3. Pipes

Unidirectional FIFO communication channel.

**Ordinary Pipes:** Parent-child communication only; unidirectional; temporary.

```c
int fd[2];
pipe(fd);  // fd[0]=read end, fd[1]=write end

if (fork() == 0) {
    close(fd[1]);
    read(fd[0], buffer, size);   // Child reads
} else {
    close(fd[0]);
    write(fd[1], "Hello", 5);    // Parent writes
}
```

**Named Pipes (FIFO):** Any processes can communicate; bidirectional possible; persists in file system.

```bash
mkfifo myfifo
echo "Hello" > myfifo   # Process 1 writes
cat < myfifo             # Process 2 reads
```

---

## 📖 4. Message Queues

Queue of typed messages in the kernel. Messages survive process termination. Receiver can select by message type.

```c
int msgid = msgget(key, IPC_CREAT | 0666);
msgsnd(msgid, &message, size, 0);       // Send
msgrcv(msgid, &message, size, type, 0); // Receive
```

---

## 📖 5. Sockets

Communication **endpoints** for network communication.

- **Stream sockets (TCP):** Reliable, ordered
- **Datagram sockets (UDP):** Unreliable, fast

**Use:** Client-server, distributed systems

---

## 📖 6. Signals

Software interrupts for **event notifications**.

| Signal | Meaning |
|--------|---------|
| `SIGKILL` | Terminate process (cannot be caught) |
| `SIGSTOP` | Stop process |
| `SIGINT` | Interrupt (Ctrl+C) |

```c
kill(pid, SIGTERM);               // Send signal
signal(SIGINT, handler_function); // Handle signal
```

---

## 📝 Quiz

**Q20.** Two processes on the same machine need to exchange 500 MB of data. Which IPC mechanism is most suitable?
> **Shared Memory** — it avoids copying data through the kernel. At 500 MB, the overhead of message passing (copying through kernel on every operation) would be severe. After initial setup, shared memory access is as fast as reading a local variable.

# 18. Shared Memory vs Message Passing

## 📖 Detailed Comparison

| Feature | Shared Memory | Message Passing |
|---------|--------------|-----------------|
| **Speed** | ⚡ Fast | 🐌 Slow |
| **Kernel involvement** | Only during setup | Every send/receive |
| **Synchronization** | Programmer's responsibility | OS handles it |
| **Data size** | Large data — very efficient | Small messages — better |
| **Complexity** | More complex (manual sync) | Simpler |
| **Protection** | Minimal | Better |
| **Distributed systems** | Same machine only | Works across machines |
| **Race conditions** | Possible | No |
| **Example** | Database shared buffers | Email, RPC, chat |

---

## 📖 When to Use Which

**Use Shared Memory when:**
- Processes are on the **same machine**
- Exchanging **large amounts** of data
- **High-speed** communication is needed
- You are willing to handle synchronization manually

**Use Message Passing when:**
- Building **distributed** systems (processes on different machines)
- Messages are **small**
- You want the **OS to handle synchronization**
- You need **protection** between processes

```c
// SHARED MEMORY (fast, direct)

// MESSAGE PASSING (safe, via kernel)

```

# 19. Remote Procedure Call (RPC)

## 📖 What is RPC?

A mechanism allowing a program to **execute a procedure on another computer** as if it were a local function call.

**Key Idea:** Make network communication look like **normal function calls** — hide all network details.

---

## 📖 Without RPC vs With RPC

```c
// Without RPC (manual network programming):
// 1. Open socket
// 2. Marshal data (convert to bytes)
// 3. Send data over network
// 4. Wait for response
// 5. Unmarshal response
// 6. Close socket

// With RPC (just call the function!):
result = add_on_server(5, 3);
// All network details handled transparently by RPC
```

---

## 📖 RPC Components

```
Client Machine                              Server Machine
┌─────────────┐                           ┌─────────────┐
│   Client    │                           │   Server    │
│  Program    │                           │  Program    │
└──────┬──────┘                           └──────▲──────┘
       │ Local call                              │ Execute
       ▼                                         │
┌─────────────┐                           ┌─────────────┐
│   Client    │                           │   Server    │
│    Stub     │                           │    Stub     │
└──────┬──────┘                           └──────▲──────┘
       │ Marshal                                 │ Unmarshal
       ▼                                         │
┌─────────────┐         Network           ┌─────────────┐
│ RPC Runtime │◄─────────────────────────►│ RPC Runtime │
└─────────────┘        Messages           └─────────────┘
```

---

## 📖 How RPC Works — Step by Step

| Step | Side | Action |
|------|------|--------|
| 1 | Client | Calls `add(5, 3)` — looks like a local call |
| 2 | Client Stub | **Marshaling** — packs function name + params into byte stream |
| 3 | RPC Runtime (Client) | Sends message over network (TCP/UDP) |
| 4 | RPC Runtime (Server) | Receives message, passes to server stub |
| 5 | Server Stub | **Unmarshaling** — extracts function name and parameters |
| 6 | Server | Executes `add(5, 3)` → returns 8 |
| 7 | Return path | Server stub marshals result → sent back → client stub unmarshals → client receives 8 |

> **Marshaling:** Converting parameters to a transmittable byte stream (handles different architectures — big-endian vs little-endian).

---

## 📖 RPC Characteristics

| Characteristic | Description |
|---------------|-------------|
| **Transparency** | Looks like a local call — programmer ignores network |
| **Synchronous** | Client blocks until response arrives |
| **Binding** | Client finds server via naming service / registry |
| **IDL** | Interface Definition Language — allows different languages/platforms to interoperate |

---

## 📖 RPC Issues and Semantics

**Failure scenarios:** Network failure, server crash, lost message.

**Solutions:** Timeouts, retries, idempotent operations.

**Execution Semantics:**

| Semantic | Meaning | Safety |
|---------|---------|--------|
| At-most-once | Execute 0 or 1 time | Safe — no duplicates |
| At-least-once | Execute 1 or more times | May duplicate |
| Exactly-once | Execute exactly once | Ideal but hardest to guarantee |

---

## 📖 Popular RPC Systems

| System | Notes |
|--------|-------|
| **gRPC** (Google) | Modern, uses Protocol Buffers |
| **XML-RPC** | Uses XML for messages |
| **JSON-RPC** | Uses JSON |
| **Java RMI** | Java Remote Method Invocation |
| **CORBA** | Common Object Request Broker Architecture |

---

## 📖 Advantages and Disadvantages

| Advantages | Disadvantages |
|-----------|---------------|
| Abstraction — hides network complexity | Slower than local calls |
| Portability — language/platform independent (with IDL) | Network failures complicate behavior |
| Simplicity — looks like local function call | Harder to debug distributed systems |
| Reusability — share services across apps | Security risks (network attacks) |

---

## 📝 Quiz

**Q22.** What is the role of the Client Stub in RPC?
> The Client Stub acts as a **proxy** for the remote function. When the client calls the remote procedure, the stub intercepts the call, **marshals** (serializes) the parameters into a byte stream, and passes it to the RPC runtime for transmission. It makes the remote call look local.

# 20. Critical Section Problem

## 📖 What is a Critical Section?

A part of code where **shared resources** (variables, files, devices) are accessed.

```c
int counter = 0;  // Shared variable

void increment() {
    // ENTRY SECTION — request permission
    counter++;     // ← CRITICAL SECTION (shared resource access)
    // EXIT SECTION — release
}
```

**Problem:** If two processes execute the critical section **simultaneously**, data becomes **inconsistent**.

---

## 📖 Critical Section Structure

```c
do {
    // 1. ENTRY SECTION
    //    Request permission to enter critical section

    // 2. CRITICAL SECTION
    //    Access shared resources

    // 3. EXIT SECTION
    //    Release critical section

    // 4. REMAINDER SECTION
    //    Other non-critical code

} while (true);
```

---

## 📖 Three Requirements for a Valid Solution

> **Memory Trick: "MPB"** → **M**utual Exclusion, **P**rogress, **B**ounded Waiting

### 1. Mutual Exclusion 🔒

**Only ONE process** can execute in the critical section at a time.

```
Correct:    P1: [CS]  wait  wait
            P2:  wait  [CS]  wait    ← One at a time ✓

Violation:  P1: [CS]  [CS]
            P2: [CS]  [CS]           ← Both in CS simultaneously ✗
```

### 2. Progress ⏩

If **no process** is in the CS and some want to enter, selection **cannot be postponed indefinitely** — decision must be made in **finite time**. (No deadlock.)

```
CS is empty. P1 wants to enter. P2 wants to enter.
→ A decision MUST be made (select P1 or P2) — cannot wait forever.

Violation: P1 waiting for P2's signal, P2 waiting for P1's — both wait forever ✗
```

### 3. Bounded Waiting ⏱️

After a process requests entry, there is a **limit** on how many times other processes can enter the CS before it gets its turn. (No starvation.)

```
P1 requests at t=0.
Maximum: P2 enters once, P3 once, ... Pn once → then P1 MUST get CS.
P1 enters within (n-1) turns ✓

Violation: P2 keeps entering while P1 waits forever ✗
```

---

## 📖 Summary of Requirements

```
┌─────────────────────────────────────┐
│   CRITICAL SECTION REQUIREMENTS     │
├─────────────────────────────────────┤
│ 1. MUTUAL EXCLUSION                 │
│    → Only one process in CS         │
│    → Safety / Correctness           │
│                                     │
│ 2. PROGRESS                         │
│    → No deadlock                    │
│    → Selection in finite time       │
│    → Liveness                       │
│                                     │
│ 3. BOUNDED WAITING                  │
│    → No starvation                  │
│    → Fairness guaranteed            │
│    → Everyone gets a turn           │
└─────────────────────────────────────┘
```

---

## 📖 Peterson's Algorithm — A Classic Solution

```c
// Shared variables
int turn;
bool flag[2];

// Process Pi (other process is Pj):
do {
    flag[i] = true;                    // I want to enter
    turn = j;                          // Give turn to other

    while (flag[j] && turn == j);      // Wait if j wants in and it's j's turn

    // ── CRITICAL SECTION ──

    flag[i] = false;                   // I'm done

    // ── REMAINDER SECTION ──
} while (true);
```

**This satisfies all three requirements** (Mutual Exclusion ✓, Progress ✓, Bounded Waiting ✓).

---

## 📝 Quiz

**Q23.** Peterson's Algorithm: If both P0 and P1 set their flags simultaneously and both set `turn`, what happens?
> The second assignment to `turn` wins (one overwrites the other). If `turn = 0` last, then P0 gets priority. If `turn = 1` last, P1 gets priority. The `while` condition ensures only one enters. This guarantees **mutual exclusion** and **progress**.

# 21. Process Synchronization – Need and Tools

## 📖 What is Process Synchronization?

Coordination of execution of **multiple processes** to ensure:
- Data consistency
- Proper order of execution
- No race conditions

---

## 📖 Why Synchronization is Needed

### Problem 1: Race Condition — Bank Account

```
Initial balance: $100

Process P1 (Deposit $50):      Process P2 (Withdraw $20):
1. Read balance: $100           1. Read balance: $100
2. Add 50 = $150                2. Subtract 20 = $80
3. Write balance: $150          3. Write balance: $80

Expected: $130    Actual: $80 (or $150)!

Time | P1 Action       | P2 Action       | Balance
-----|-----------------|-----------------|--------
 t1  | Read ($100)     |                 | $100
 t2  |                 | Read ($100)     | $100
 t3  | Compute ($150)  |                 | $100
 t4  |                 | Compute ($80)   | $100
 t5  | Write ($150)    |                 | $150
 t6  |                 | Write ($80)     | $80

Lost update! The $50 deposit disappeared.
```

### Problem 2: Producer-Consumer — count++ is NOT Atomic

```assembly
; count++ translates to THREE operations:
LOAD  register ← count
ADD   register = register + 1
STORE count ← register

; Two processes interleaved on count=5:
P1: LOAD R1=5
              P2: LOAD R2=5
P1: R1=6
              P2: R2=4
P1: STORE count=6
              P2: STORE count=4

Expected: 5    Actual: 4 or 6 — unpredictable!
```

---

## 📖 Five Reasons Synchronization is Needed

| Reason | Problem Prevented |
|--------|------------------|
| **Data Consistency** | Lost updates, corrupted data |
| **Order of Execution** | Consumer before producer is ready |
| **Resource Allocation** | Two processes printing simultaneously |
| **Deadlock Prevention** | Processes waiting for each other |
| **Program Correctness** | Predictable, repeatable results |

---

## 📖 Synchronization Tools

```
┌─────────────────────────────────────┐
│       SYNCHRONIZATION TOOLS         │
├─────────────────────────────────────┤
│  1. Locks (Mutex)                   │
│  2. Semaphores                      │
│  3. Monitors                        │

│  5. Atomic Operations               │

```

## 📝 Quiz

**Q24.** Is `x = x + 1` in a high-level language an atomic operation?
> **No.** Even a simple increment compiles to multiple machine instructions (load, add, store). Any of these steps can be interrupted between processes. This is why synchronization mechanisms are needed even for simple shared variable updates.

# 22. Race Conditions

## 📖 What is a Race Condition?

A situation where **multiple processes** access **shared data** concurrently, and the **final result depends on the order of execution**.

---

## 📖 Example 1: Counter Increment

```c
int counter = 0;  // Shared

// P1 and P2 both run:
for (int i = 0; i < 1000; i++)
    counter++;

// Expected: 2000    Actual: Anything from 2 to 2000!
```

**Why? — counter++ is 3 operations:**

```
Time | P1              | P2              | counter
-----|-----------------|-----------------|--------
 t1  | LOAD R1 (0)     |                 |    0
 t2  |                 | LOAD R2 (0)     |    0
 t3  | R1 = R1+1 (1)   |                 |    0
 t4  |                 | R2 = R2+1 (1)   |    0
 t5  | STORE R1 (1)    |                 |    1
 t6  |                 | STORE R2 (1)    |    1

Result: 1  (should be 2!)  — one increment lost.
```

---

## 📖 Example 2: Bank Account

```c
int balance = 1000;

// T1 — Deposit $500:               // T2 — Withdraw $200:
int temp = balance;  // Read         int temp = balance;  // Read
temp = temp + 500;   // Modify       temp = temp - 200;   // Modify
balance = temp;      // Write        balance = temp;      // Write
```

**Sequential (correct):** 1000 → 1500 → 1300 ✓

**Interleaved (race):**
```
Time | T1                  | T2                  | balance
-----|---------------------|---------------------|--------
 t1  | Read (1000)         |                     | 1000
 t2  |                     | Read (1000)         | 1000
 t3  | Compute (1500)      |                     | 1000
 t4  |                     | Compute (800)       | 1000
 t5  | Write (1500)        |                     | 1500
 t6  |                     | Write (800)         |  800

Final: $800  ✗  (Lost the $500 deposit!)
```

---

## 📖 Example 3: Print Spooler

```c
int next_free_slot = 0;
char *files[100];

// Process A:                     // Process B:
files[next_free_slot] = "A.txt";  files[next_free_slot] = "B.txt";
next_free_slot++;                  next_free_slot++;
```

```
Time | Process A              | Process B
-----|------------------------|------------------------
 t1  | Read next_slot (7)     |
 t2  |                        | Read next_slot (7)
 t3  | files[7] = "A.txt"     |
 t4  |                        | files[7] = "B.txt"  ← Overwrites A!
 t5  | next_slot = 8          |
 t6  |                        | next_slot = 8

Result: A.txt is lost. Both stored at slot 7. Slot 8 empty.
```

---

## 📖 Conditions for a Race Condition

All **three** must hold simultaneously:

1. **Shared data** — multiple processes access the same data
2. **Concurrent access** — processes access it simultaneously
3. **At least one write** — at least one process modifies the data

---

## 📖 Prevention

```c
// Without synchronization (race condition):
counter++;

// With lock (safe):
acquire_lock(&lock);
counter++;           // Only one process here at a time
release_lock(&lock);
```

---

## 📝 Quiz

**Q25.** Two processes only **read** a shared variable — can a race condition occur?
> **No.** A race condition requires at least one **write** operation. If all processes are only reading, the shared data cannot be corrupted regardless of execution order.

# 23. Semaphores

## 📖 What is a Semaphore?

A **synchronization tool** — an integer variable accessed only through two **atomic operations**.

**Invented by:** Edsger Dijkstra

| Operation | Also called | Effect |
|-----------|-------------|--------|
| `wait(S)` | P(S), down(S) | Decrement S; block if S ≤ 0 |
| `signal(S)` | V(S), up(S) | Increment S; wake a waiting process |

> **Critical:** These operations are **ATOMIC** (indivisible — cannot be interrupted mid-operation).

---

## 📖 Basic Operations

```c
// wait() — acquire
wait(S) {
    while (S <= 0);  // Busy wait (spinlock)
    S--;
}

// signal() — release
signal(S) {
    S++;
}
```

---

## 📖 Types of Semaphores

### 1. Binary Semaphore (Mutex)

**Value:** 0 or 1 only. Used for **mutual exclusion** (like a lock).

```c
semaphore mutex = 1;   // Initially 1 (unlocked)

// Any process:
wait(mutex);           // Decrement to 0 — lock acquired
// ── CRITICAL SECTION ──
signal(mutex);         // Increment to 1 — lock released

// Example with two processes:
Initial: mutex = 1

P1: wait(mutex)  → mutex = 0 (P1 enters CS)
    ...CS...
P1: signal(mutex) → mutex = 1

P2: wait(mutex)  → If P1 in CS, mutex=0, P2 blocks
                 → When P1 signals, mutex=1, P2 enters
```

---

### 2. Counting Semaphore

**Value:** Any non-negative integer. Used for **resource allocation** (N identical resources).

```c
semaphore resource = 5;   // 5 resources available

wait(resource);    // Acquire 1 resource (decrement)
// Use resource
signal(resource);  // Release resource (increment)
```

**Example — Printer Queue (3 printers):**

```
semaphore printers = 3;

P1: wait(printers);  // printers = 2
P2: wait(printers);  // printers = 1
P3: wait(printers);  // printers = 0
P4: wait(printers);  // printers = 0 → P4 BLOCKS (all busy)

P1: signal(printers); // printers = 1 → P4 unblocks
```

---

## 📖 Block/Wakeup Implementation (No Busy Waiting)

**Problem:** Busy waiting wastes CPU.

**Solution:** Put the process to sleep instead of spinning.

```c
typedef struct {
    int value;
    struct process *list;  // Waiting queue
} semaphore;

wait(semaphore *S) {
    S->value--;
    if (S->value < 0) {
        add_to_queue(S->list, current_process);
        block();  // Sleep — don't busy wait
    }
}

signal(semaphore *S) {
    S->value++;
    if (S->value <= 0) {
        process *P = remove_from_queue(S->list);
        wakeup(P);
    }
}
```

---

## 📖 Semaphore for Critical Section

```c
semaphore mutex = 1;

do {
    wait(mutex);

    // ── CRITICAL SECTION ──

    signal(mutex);

    // ── REMAINDER SECTION ──
} while (true);
```

**Guarantees:** ✅ Mutual exclusion, ✅ Progress, ✅ Bounded waiting (with queue)

---

## 📖 Producer-Consumer with Semaphores

```c
#define N 10

semaphore mutex = 1;   // Mutual exclusion on buffer
semaphore empty = N;   // Count of empty slots
semaphore full  = 0;   // Count of full slots

int buffer[N];

void producer() {
    while (true) {
        int item = produce();
        wait(empty);    // Wait for empty slot
        wait(mutex);    // Lock buffer
        buffer[in] = item;
        in = (in + 1) % N;
        signal(mutex);  // Unlock buffer
        signal(full);   // Signal: one more full slot
    }
}

void consumer() {
    while (true) {
        wait(full);     // Wait for full slot
        wait(mutex);    // Lock buffer
        int item = buffer[out];
        out = (out + 1) % N;
        signal(mutex);  // Unlock buffer
        signal(empty);  // Signal: one more empty slot
        consume(item);
    }
}
```

**Initial state:** mutex=1, empty=10, full=0

> ⚠️ **Order matters!** Always `wait(empty/full)` BEFORE `wait(mutex)`. Reversing the order causes **deadlock**.

---

## 📖 Advantages and Disadvantages

| Advantages | Disadvantages |
|-----------|---------------|
| Simple and elegant | Easy to make mistakes (forget wait/signal) |
| Works for any number of processes | Forgetting `wait()` → no mutual exclusion |
| Can count resources (counting semaphore) | Forgetting `signal()` → deadlock |
| No busy waiting (with block/wakeup) | Not type-safe; hard to debug |
| Portable — works on any system | |

---

## 📖 Common Semaphore Errors

**1. Deadlock:**
```c
// P1:                // P2:
wait(S);             wait(Q);
wait(Q); ← blocks   wait(S); ← blocks

// P1 holds S, P2 holds Q → Deadlock!
```

**2. Starvation:**
```
High-priority processes keep getting semaphore
Low-priority process never gets it
```

**3. Priority Inversion:**
```
Low-priority holds semaphore
High-priority waits
Medium-priority preempts low-priority
→ High-priority waits for low-priority, which can't run!
```

---

## 📖 Quick Reference

```
BINARY SEMAPHORE (Mutex):
  Initial value : 1
  Use           : Mutual exclusion
  Analogy       : Lock / Unlock

COUNTING SEMAPHORE:
  Initial value : N (number of resources)
  Use           : Resource allocation
  Analogy       : Resource counter

OPERATIONS (must be ATOMIC!):
  wait(S)   / P(S) : Decrement; block if needed
  signal(S) / V(S) : Increment; wake if needed

PRODUCER-CONSUMER:
  3 semaphores needed: mutex, empty, full
  Order: wait(empty/full) BEFORE wait(mutex)
```

---

## 📝 Quiz

**Q26.** A semaphore `S` has value −3. What does this mean?
> `S = −3` means there are **3 processes currently blocked** (sleeping) waiting for this semaphore. With the block/wakeup implementation, a negative value represents the number of waiting processes.

# 25. Quick Reference – Formulas & Tables

## 🔑 Scheduling Formulas

```
Turnaround Time (TAT) = Completion Time (CT) − Arrival Time (AT)
                      = Waiting Time (WT) + Burst Time (BT)

Waiting Time (WT)     = TAT − BT
                      = CT − AT − BT

Response Time (RT)    = First CPU Allocation − AT

Average TAT           = Σ TAT / Number of processes
Average WT            = Σ WT  / Number of processes

CPU Utilization       = (Total CPU busy time / Total time) × 100%

Throughput            = Processes completed / Total time
```

---

## 🔑 Semaphore Quick Reference

```
BINARY SEMAPHORE:
  Initial value = 1
  wait(S)   → S-- ; block if S < 0
  signal(S) → S++ ; wake if S ≤ 0

COUNTING SEMAPHORE:
  Initial value = N (resource count)
  Negative value = |S| processes waiting

PRODUCER-CONSUMER:
  mutex = 1, empty = N, full = 0
  Producer: wait(empty) → wait(mutex) → produce → signal(mutex) → signal(full)
  Consumer: wait(full)  → wait(mutex) → consume → signal(mutex) → signal(empty)
```
